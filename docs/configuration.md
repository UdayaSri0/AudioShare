# Configuration, Persistence, And Recovery

This document describes the current configuration and recovery model in
SynchroSonic. The goal is a versioned, understandable settings layer that
preserves user choices without hiding when the app had to repair or discard a
saved configuration.

## Config File Locations

By default SynchroSonic uses XDG-style directories on Linux:

- config directory: `$XDG_CONFIG_HOME/synchrosonic` or `~/.config/synchrosonic`
- state directory: `$XDG_STATE_HOME/synchrosonic` or
  `~/.local/state/synchrosonic`
- active config file: `config.toml`
- portable import/export file: `config-export.toml`
- structured log file: `app-log.jsonl`

Two environment variables override the base directories:

- `SYNCHROSONIC_CONFIG_DIR`
- `SYNCHROSONIC_STATE_DIR`

When those variables are set, SynchroSonic still creates its own
`synchrosonic` subdirectory underneath the provided root.

## Config Schema

The current schema version is `1`.

Top-level sections:

- `schema_version`
- `audio`
- `discovery`
- `transport`
- `receiver`
- `ui`
- `diagnostics`

Important saved settings in this version:

- `audio.preferred_source_id`
- `audio.local_playback_enabled`
- `audio.local_playback_target_id`
- `transport.quality`
- `transport.target_latency_ms`
- `receiver.start_on_launch`
- `receiver.playback_target_id`
- `receiver.latency_preset`
- `ui.prefer_dark_theme`
- `ui.last_view_name`
- `diagnostics.verbose_logging`

This covers the main persistence goals for the current UI: selected audio
source, local mirror preferences, receiver playback target, receiver
auto-start, latency preset, and basic UI preferences.

## Validation And Repair

Saved configuration is validated before it is applied.

Current repair rules include:

- zero or missing sample rate, channel count, capture buffer size, or capture
  latency fall back to defaults
- empty discovery service type or zero stale timeout fall back to defaults
- empty transport bind host, zero stream port, zero target latency, or very low
  timeout/heartbeat values fall back to defaults
- empty receiver name/bind host or zero listen port fall back to defaults
- empty `ui.last_view_name` falls back to `dashboard`

If the file schema version is older than the current version, SynchroSonic
repairs the structure in memory, upgrades `schema_version`, and writes the
repaired config back to disk.

If the file schema version is newer than the application supports, the app does
not guess. It treats the config as unsupported, backs it up, and falls back to
defaults.

## Startup And Autosave Behavior

On startup, the app:

1. Resolves config and state paths.
2. Loads `config.toml` if present.
3. Repairs and re-saves it if validation finds recoverable issues.
4. Creates a default config if no file exists.
5. Falls back to defaults if the file cannot be parsed or is unsupported.
6. Restores runtime state from the saved configuration.

Most preference changes made in the GTK UI are saved immediately back to the
active config file. Import and export are intentionally simple:

- export writes the current active config to `config-export.toml`
- import reads `config-export.toml`, validates it, applies what can be applied
  safely, and then saves it as the new active config

## Structured Logging

Structured logging is enabled through `tracing` and written as JSON lines to
`app-log.jsonl`.

Each log entry includes:

- `timestamp_unix_ms`
- `level`
- `target`
- `message`
- structured `fields`
- optional source `file`
- optional source `line`

The diagnostics page shows:

- recent diagnostic events generated by app state and recovery logic
- recent structured log entries from the in-memory log store
- the on-disk structured log file location

When the app starts, it hydrates the in-memory log viewer from the tail of the
existing JSON-lines log file when possible.

## Recovery Behavior

### App restart

- saved audio source, local playback target, receiver target, latency preset,
  theme preference, verbose logging preference, and last visited page are
  restored from config
- receiver mode can optionally auto-start after configuration restore when
  `receiver.start_on_launch = true`

### Receiver disconnect

- the receiver runtime clears its active transport/buffer state and returns to
  listening mode instead of keeping stale playback state around
- diagnostics and structured logs record the disconnect reason and whether
  reconnect behavior was suggested

### Selected device disappearance

- if the saved or selected capture source disappears, SynchroSonic falls back
  to another available source and records the change in diagnostics
- if a selected playback target disappears, the selection is preserved in
  config and shown as unavailable rather than being silently erased
- if the selected discovered receiver disappears, the app reconciles to another
  valid receiver when available and reports the change in diagnostics

### Invalid saved configuration

- parse failures, unsupported schema versions, or otherwise unusable config
  files trigger a fallback to defaults
- the invalid file is renamed to `config.invalid-<unix-seconds>.toml` when
  possible
- diagnostics explain that the app recovered with defaults rather than silently
  ignoring the problem

## Current Limitations

This is intentionally a practical first version.

- import/export uses a fixed portable file path instead of a file chooser
- verbose logging preference takes effect on the next launch, not immediately
- schema migration is still repair-based; there is not yet a dedicated
  multi-version migration framework
- recovery preserves selected playback targets when they disappear, but active
  audio sessions still depend on the underlying backend being able to recover
  cleanly

## Future Improvements

- add explicit schema migrators when version `2+` introduces shape changes
- add rotation/retention policies for structured log files
- allow user-selected import/export destinations
- expose more recovery history and persistence status in diagnostics
